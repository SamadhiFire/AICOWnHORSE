- 首次生成入口：[src/services/practice-generation-service.ts](d:\桌面\AGithub\AI-\学习软件\src\services\practice-generation-service.ts)
- 追加批次入口（兼容保留）：[src/utils/generation-job.ts](d:\桌面\AGithub\AI-\学习软件\src\utils\generation-job.ts)
- 本地生成编排入口：[src/utils/pipeline.ts](d:\桌面\AGithub\AI-\学习软件\src\utils\pipeline.ts) -> [src/utils/quiz-generation/orchestrator.ts](d:\桌面\AGithub\AI-\学习软件\src\utils\quiz-generation\orchestrator.ts)

## 现在的 source of truth

- 最终目标 source of truth：后端 `/questions/generate`
- 当前过渡态 source of truth：前端统一先走后端生成接口；若后端接口未实现或不可用，再显式切到本地 `local_fallback`
- 关键变化：页面层不再决定“先生成一部分再补生成”，而是默认一次性请求整套题目

## 当前链路结论

过去的真实链路是“首批题 + 后续批次追加”。  
现在前端默认行为已经改为“一次性生成所有题目”：

1. 首页提交时让 `initialBatchCount = targetCount`
2. 本地 fallback 也一次性按 `targetCount` 生成
3. 如果首次返回已经达到目标题量，就不再保留运行中的 `generationJob`
4. 旧的 `generation-job` / `batch` 逻辑仅作兼容保留，不再是默认主路径

## 过渡策略

1. 前端统一只发起一次“整套题目”请求。
2. 后端若已支持一次性生成，应直接返回完整 `session.questions`，并把 `generationJob` 标记为 `completed`，或至少保证 `loadedCount >= targetCount`。
3. 只有当后端仍保留旧批次实现、且首次未返回足量题目时，前端才会落回兼容批次逻辑。
4. 等后端完全稳定后，可以进一步删除前端的批次触发代码，把前端彻底收敛成“展示 + 状态流转 + 结果承接”。
